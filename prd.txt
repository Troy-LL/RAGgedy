1. Overview

1.1 Purpose

This document defines the product requirements for the 01_Naive_RAG module of the RAGgedy open-source repository. It covers the Edu-Scholar pipeline — a production-ready Retrieval-Augmented Generation system designed for educational content, with a primary focus on College Entrance Test (CET) preparation and local skills training materials.

This module serves a dual purpose: it is both a functional pipeline that developers can drop into their own projects, and a guided learning environment for developers who want to understand how RAG systems work from first principles.

1.2 Background

Most open-source RAG repositories provide a single monolithic example that assumes the developer already has their own data and understands the underlying architecture. RAGgedy takes a different approach: it provides a “Lego kit” of modular, clearly structured templates ordered by architectural complexity, each bundled with datasets and evaluation tooling.

The 01_Naive_RAG module is the entry point — the simplest possible RAG pipeline that still demonstrates every core concept. It is intentionally incomplete so that learners can see exactly what breaks when they push it harder, motivating the need for the techniques introduced in 02_Advanced_RAG.

1.3 Scope

This PRD covers the following:

Folder structure and file responsibilities for 01_Naive_RAG

Technology stack and dependency requirements

The Edu-Scholar starter dataset specification

The ingestion pipeline (ingest.py)

The query pipeline (query.py)

Configuration management (config.py)

Evaluation setup using Ragas

The guided learning notebook

The following are explicitly out of scope for this module:

Hybrid search / BM25 retrieval (covered in 02_Advanced_RAG)

Cross-encoder reranking (covered in 02_Advanced_RAG)

Agentic workflows (covered in 03_Agentic_RAG)

Geographic / spatial indexing (covered in 04_Geo_RAG)

2. Goals & Non-Goals

2.1 Goals

Provide a fully working, locally-runnable RAG pipeline with zero proprietary API dependencies.

Give learners a concrete starting point with bundled sample data so they can run the pipeline within minutes of cloning the repo.

Make every architectural decision visible and explainable — configuration is centralized, pipeline stages are separated into distinct scripts, and every component has a documented reason for existing.

Establish a Ragas baseline score that subsequent modules (02, 03, 04) can demonstrably improve upon.

Include an intentionally degraded “broken” variant that shows learners what happens when chunking and retrieval are done poorly.

2.2 Non-Goals

This module does not aim to achieve state-of-the-art retrieval performance. It is a teaching tool, not a production-optimized system.

This module does not support cloud-hosted models or paid embedding APIs. The entire stack must run locally via Ollama and Hugging Face.

This module does not include a web interface or REST API. Interaction is via CLI and Jupyter notebooks only.

Dependency management is modular: each folder in RAGgedy maintains its own requirements.txt.

5. System Architecture

5.1 Pipeline Overview

The Naive RAG pipeline consists of two distinct phases: an offline ingestion phase that builds the vector index, and an online query phase that retrieves context and generates answers.

Pipeline Flow

Document → Chunk (SentenceSplitter) → Embed (BAAI/bge-m3) → Store (ChromaDB) → Query → Retrieve (top-K cosine) → Prompt + Context → LLM (llama3) → Answer

6.2 Ingestion Pipeline (ingest.py)

The ingestion script must perform the following operations in order:

Load all documents from DATA_DIR using SimpleDirectoryReader (supports .txt and .pdf).

Print the number of documents loaded.

Chunk documents using SentenceSplitter with CHUNK_SIZE and CHUNK_OVERLAP from config.

Print the number of chunks created.

Load the embedding model specified by EMBED_MODEL. Model is downloaded on first run and cached.

Create or connect to a persistent ChromaDB client at CHROMA_DIR.

Get or create a collection named COLLECTION_NAME.

Build and persist a VectorStoreIndex from the chunked nodes.

Print a success message and instructions to run query.py.

6.3 Query Pipeline (query.py)

The query script must perform the following operations:

Load the embedding model and LLM from config settings.

Connect to the existing ChromaDB index (must fail gracefully if ingest.py has not been run).

Expose an interactive CLI loop that accepts user questions.

For each question: embed the query, retrieve the top-K chunks, construct a prompt with the retrieved context, generate a response using Ollama.

Print the answer followed by the source file names and similarity scores for each retrieved chunk.

Exit cleanly on 'quit' or 'exit' input.

6.4 Evaluation (evaluation/eval_naive.py)

The evaluation script must:

Load questions.json from the data directory.

Run each question through the query engine and collect: the generated answer, the retrieved context chunks, and the ground-truth answer.

Evaluate using Ragas with at minimum the following metrics: context_precision and faithfulness.

Print a summary table of scores.

Optionally export results to a JSON file for comparison across module versions.

Dataset Design Note

The dataset is synthetic but realistic. It is designed to produce mid-range Ragas scores on the naive pipeline (target: 0.55–0.70 faithfulness) — good enough to demonstrate the system works, but with clear room for improvement that the 02_Advanced_RAG module addresses.

8. Learning Experience Requirements

8.1 Guided Notebook (01_walkthrough.ipynb)

The notebook is the primary learning artifact. It must walk through the entire pipeline step by step, with explanatory commentary at each stage. It is not a duplicate of the scripts — it is a narrated tour of what the scripts do and why.

Required sections:

Introduction: what is RAG and why does it exist?

Configuration: load config.py and explain each parameter.

Document loading: show a sample document and its metadata.

Chunking: show before/after, print chunk sizes, visualize overlap.

Embedding: show the shape of a chunk's embedding vector.

Indexing: confirm ChromaDB collection was created.

Querying: run 3 example questions, show retrieved chunks with scores.

Evaluation: run Ragas and display scores.

Next steps: explain what breaks and why, link to 02_Advanced_RAG.

8.2 The Broken Variant

A second script, ingest_broken.py, must be included. It uses intentionally degraded settings to produce poor retrieval performance. Learners run both, compare Ragas scores, and develop intuition about why each parameter matters.

Parameter

Broken Value (vs. Default)

CHUNK_SIZE

4096 (vs. 512) — chunks are too large to retrieve precisely

CHUNK_OVERLAP

0 (vs. 64) — context is lost at chunk boundaries

TOP_K

1 (vs. 5) — too few chunks retrieved to answer multi-part questions

Learning Objective

After running both variants and comparing Ragas scores, learners should be able to explain in their own words why chunk size, overlap, and top-K each affect retrieval quality. This prepares them to make informed architectural decisions in their own projects.

9. Non-Functional Requirements

9.1 Performance

Ingestion of 100 passages must complete in under 5 minutes on a 2020-era laptop (Apple M1 or Intel i7 equivalent) without GPU.

Query latency must be under 30 seconds per question on the same hardware (Ollama + llama3, CPU inference).

Embedding model (BAAI/bge-m3) must be cached after first download; subsequent runs must not re-download.

9.2 Reproducibility

A single command (python ingest.py followed by python query.py) must be sufficient to run the full pipeline after cloning the repository and installing dependencies.

All random seeds must be fixed where applicable to ensure consistent chunking and evaluation results across runs.

requirements.txt must pin exact package versions.

9.3 Readability

All scripts must include inline comments explaining non-obvious decisions.

Variable names must be descriptive. No single-letter variables outside of loop counters.

Each file must begin with a docstring explaining its purpose, inputs, and outputs.

12. Future Work & Next Module

12.1 What 02_Advanced_RAG Will Add

The limitations of the naive pipeline directly motivate the next module. The following gaps should be observable from 01_Naive_RAG's Ragas scores:

Dense-only retrieval misses keyword-critical queries — 02 adds BM25 hybrid search.

Top-K retrieval returns noisy chunks — 02 adds cross-encoder reranking to re-score and filter.

No query transformation — 02 adds HyDE (Hypothetical Document Embeddings) for ambiguous questions.

12.2 Dataset Expansion

Add Filipino-language passages to stress-test bge-m3's multilingual capabilities.

Add tabular data (CSV-formatted passages) to test chunking strategies for structured content.

Add multi-document queries that require synthesizing information across multiple passages.

12.3 MCP Compatibility

Future iterations should structure the ingestion and query pipelines so that they can be exposed as MCP (Model Context Protocol) tools, enabling integration with Cursor IDE and other MCP-compatible clients. This is a design goal for v2.0 of the module, not a requirement for v1.0.

Appendix A: Glossary

Term

Definition

RAG

Retrieval-Augmented Generation. A technique that grounds LLM outputs in retrieved documents to reduce hallucination.

Chunk

A fixed-size segment of a document, created by the SentenceSplitter. The unit of retrieval.

Embedding

A high-dimensional vector representation of a text chunk that captures its semantic meaning.

ChromaDB

An open-source vector database that stores embeddings and enables similarity search.

Ollama

A local LLM runtime that serves open-source models (llama3, mistral, etc.) via a REST API.

bge-m3

BAAI's BGE-M3 embedding model. Supports 100+ languages, 8192 token context. Runs locally.

Ragas

An open-source evaluation framework for RAG pipelines. Measures faithfulness, context precision, and more.

Faithfulness

Ragas metric: does the generated answer stay grounded in the retrieved context? (0–1)

Context Precision

Ragas metric: are the retrieved chunks relevant to the question? (0–1)

CET

College Entrance Test. A standardized exam used in the Philippines for university admission.

MCP

Model Context Protocol. An emerging standard for exposing tools and data sources to LLM clients.
